Branches

Herhaling

GIT kan wijzigingen in tekst bestanden herkennen en opslaan. Op die manier is het makkelijk om bij te houden wat er veranderd is gedurende het project en ook wie die wijzigingen gemaakt heeft.

Commando's

  • git status
  • git add [file]
  • git add -m "[message]"
  • git pull
  • git push

GIT houdt die informatie bij in de .git map. (git init) Dit is in principe een soort database met alle informatie er in over de inhoud, de wijzigingen en de status van alle bestanden in de "Working Directory".
In de staging area worden alle bestanden die horen bij een wijziging verzameld. (git add [filename])
Waarna de hele wijziging als één set toegevoegd worden aan de local repository. Voorzien van het juiste commentaar. (git commit -m "Jouw commentaar"])
Daarna is er de optie om de wijziging door te sturen naar de upstream repository. (git push) Dat kan een eigen GIT server zijn, of natuurlijk github.com

Wat zijn branches?

Waarom branches

Zolang je slechts kleine wijzigingen doet in de applicatie is het gebruik van branches in principe niet nodig. Ook in deze webpagina is het misschien wat overdreven.

Branches worden onmisbaar wanneer een teamlid voor langere tijd met een grote wijziging (De term daarvoor is "feature") bezig is. Dit soort grote features worden meestal verdeeld in kleinere behapbare wijzigingen die samen een grote nieuwe mogelijkheid voor de software opleveren.

  • Toevoegen of herzien van een nieuwe database
  • Vervangen van het belichtingssysteem
  • Een nieuw scherm met alle bijbehorende knoppen en acties

Hoe werkt dat?

Alle commando's die we tot nog toe gebruikt hebben functioneren binnen één branch. Deze branch is de hoofd tak. De gebruikelijke naam daarvoor is "master" of "main".

Deze master branch heeft verder geen bijzondere status. Het is mogelijk en zelfs aan te raden om een afgebakende hoeveelheid werk uit te voeren binnen een andere branch.

Wanneer je wisselt naar een andere branch, werkt je verder op precies dezelfde manier. Totdat jouw hele werk gereed is en je alle wijzigingen in één klap toevoegt aan de hoofd ("master") branch. Deze hoofd branch wordt dan in één keer voorzien van een nieuwe feature.

Waarom branches?

De laatste reden om te werken in een branch is zodat de betrokken personen uitgebreid de tijd hebben om de nieuwe feature van de software te bouwen en te testen zonder dat de hoofd branch tijdelijk buiten gebruik is.

Door deze manier van werken blijven alle features van werken met GIT ter beschikking. Iedere dag kan een afgeronde wijziging worden gecommit en gepusht naar de centrale server. En alle leden van het sub-team kunnen die wijzigingen delen met elkaar.

Pas wanneer de verantwoordelijke groep helemaal tevreden is met de nieuwe code. Kan de gehele set van benodigde wijzigingen worden toegevoegd.

Ten slotte is het daarna mogelijk om de branch op te heffen. Dit is afhankelijk van de ontwikkel methode die het team hanteerd. (Zie ook git flow.)

GIT Branch Commando's

Start een nieuwe tak met wijzigingen

git branch [branchnaam]

Na het uitvoeren van deze instructie is er een nieuwe tak voorhanden waarin gewerkt kan worden.

Wisselen naar een andere branch

git checkout [branchnaam]

Dit commando geeft aan dat je gaat werken met de genoemde branch. Dat kan helemaal aan het begin zijn, als de branch nog leeg is. Maar het is ook mogelijk om later aan te sluiten.

In dat laatste geval worden de meest recente wijzigingen die al in de branch zijn gecommit gesynchroniseerd met jouw working directory. En kun je het werk doorzetten.

Het is ook mogelijk om terug te gaan naar de hoofd branch. In dat geval wordt jouw working directory voorzien van alle code die daar beschikbaar is. Maar verlies je de wijzigingen die alleen nog maar in de sub-branch gemaakt zijn.

Reïntegratie

git merge [branchnaam] --no-commit

Terwijl je werkt aan een sub-branch loopt de lijst van wijzigingen ten opzichte van de rest van de hoofd branch steeds verder uit elkaar. De nieuwe branch en de hoofdbranch bevatten dan steeds meer verschillen. Sommige daarvan kunnen niet automatisch overgenomen worden.

Om er voor te zorgen dat jouw branch en alle wijzigingen die daarin zitten zoveel mogelijk "compatible" moet je dagelijks het pull commando blijven uitvoeren.
